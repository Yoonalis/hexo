方法名 功能说明 注意

3 JMM 内存模型

java内存模型规定所有变量保存在主内存中，每个线程有自己的工作内存。线程的工作内存中保存了该线程使用到的变量的主内存拷贝，线程对变量的所有操作都必须在工作内存中进行，而不能直接读写主内存中的变量。不同的线程之间无法直接访问对方的工作内存，需通过主内存才能进行变量的传递。

4 可见性、有序性、原子性、volatile

可见性

可见性指的是当一个线程修改了共享变量后，其他线程能够立即得知这个修改，volatile可以保证可见性

有序性

有序性指的是程序按照代码的先后顺序执行

为了性能优化，编译器和处理器会可能进行指令重排序，有时候会改变程序中语句的先后顺序

原子性

原子性指的是一个或者多个操作在 CPU 执行的过程中不被中断的特性

在多线程情况下，线程会被操作系统调度进行任务切换，占有CPU时间片段的就执行，否则就阻塞

volatile

指令重排序

jvm 在不改变运行结果的前提下，对指令执行的顺序进行调整，以提高程序运行的效率

内存屏障

可见性

  1. 写屏障（sfence）保证在该屏障之前的，对共享变量的改动，都同步到主存当中
  2. 而读屏障（lfence）保证在该屏障之后，对共享变量的读取，加载的是主存中最新数据

有序性

  1. 写屏障会确保指令重排序时，不会将写屏障之前的代码排在写屏障之后
  2. 读屏障会确保指令重排序时，不会将读屏障之后的代码排在读屏障之前

volatle 原理

volatile 的底层实现原理是内存屏障，Memory Barrier``（Memory Fence）

对 volatile 变量的写指令后会加入写屏障

对 volatile 变量的读指令前会加入读屏障

6 synchronized 原理

6.1 monitor 原理

每个 Java 对象都可以关联一个 Monitor 对象，如果使用 synchronized 给对象上锁（重量级）之后，该对象头的 Mark Word 中就被设置指向 Monitor 对象的指针

Monitor 结构

刚开始 Monitor 中 Owner 为 null

当 Thread-2 执行 synchronized(obj) 就会将 Monitor 的所有者 Owner 置为 Thread-2，Monitor中只能有一 个 Owner

在Thread-2 上锁的过程中，如果 Thread-3，Thread-4，Thread-5 也来执行 synchronized(obj)，就会进入 EntryList BLOCKED

Thread-2 执行完同步代码块的内容，然后唤醒 EntryList 中等待的线程来竞争锁，竞争的时是非公平的

图中 WaitSet 中的 Thread-0，Thread-1 是之前获得过锁，但条件不满足进入 WAITING 状态的线程，后面讲 wait-notify 时会分析

6.2 轻量级锁

如果一个对象虽然有多线程要加锁，但加锁的时间是错开的（也就是没有竞争），那么可以 使用轻量级锁来优化。

在线程的栈桢中创建锁记录（Lock Record）对象，让锁记录中 Object reference 指向锁对象，并尝试用 cas 替换 Object 的 Mark Word，将Mark Word 的值存 入锁记录

如果 cas 替换成功

对象头中存储了锁记录地址和状态 00 ，表示由该线程给对象加锁，这时图示如下

如果 cas 失败，有两种情况

如果是其它线程已经持有了该 Object 的轻量级锁，这时表明有竞争，进入锁膨胀过程

如果是自己执行了 synchronized 锁重入，那么再添加一条 Lock Record 作为重入的计数

当退出 synchronized 代码块（解锁时）

如果有取值为 null 的锁记录，表示有重入，这时重置锁记录，表示重 入计数减一

6.3 偏向锁

如果开启了偏向锁（默认开启），那么对象创建后，markword 最后 3 位为 101，否则为001

可以添加VM 参数 -XX:-UseBiasedLocking 禁用偏向锁

注意

处于偏向锁的对象解锁后，线程 id 仍存储于对象头中

正常状态对象一开始是没有 hashCode 的，第一次调用才生成

调用hashcode()方法，会导致偏向锁被撤销(存储hashcode)

其他线程使用对象，会导致偏向锁被撤销（轻量级锁）

调用wait/notify，会导致偏向锁被撤销（重量级锁）

轻量级锁会在锁记录中记录 hashCode

重量级锁会在 Monitor 中记录 hashCode

6.4 synchronized 锁升级

  1. 如果开启了偏向锁（默认开启），最后三位是101，处于可偏向状态，Mark Word记录线程Id

  2. 当线程A获取锁时，会对比Mark Word中的线程id和自己的线程id。

    如果相同，获取锁成功

    如果不同，他会检查Mark Word 线程id对应的线程是否存活

    ​ 如果存活，取消偏向状态，升级为轻量级锁

    ​ 如果不存活，使用cas替换Mark Word中的线程id为自己的线程id

  3. 偏向锁状态抢锁失败，代表当前锁有一定的竞争，偏向锁将升级为轻量级锁。JVM会在当前线程的线程栈中创建一条锁记录，锁记录的Object 指针指向锁对象，使用cas操作保存Mark Word的信息，同时使用cas 操作使Mark Word 指向锁记录，开启自旋，重复尝试。

  4. 当替换失败达到一定次数时(默认是10)，就会进入锁膨胀，升级为重量级锁。如果在此期间还有其他线程竞争锁，立即升级为重量级锁。

  5. 升级重量级锁， 首先申请一个Monitor对象，让Mark Word指向Monitor 对象，把加锁失败的线程放入到Monitor 对象的EntryList中阻塞

7 wait、notify、notifyAll、join原理

wait、notify、notifyAll

当Owner 线程发现条件不满足时，调用wait 方法进入到 Monitor 的 waitSet 进行等待
当调用notify 或 notifyAll 会唤醒waitSet 中等待的线程进入entryList 竞争锁

join

t1.join() 将自己加入到 t1 的waitSet 直到 t1 结束

8 park unPark 原理

每个线程都有自己的一个 Parker 对象，由三部分组成 _counter ， _cond 和 _mutex

  1. 当前线程调用 Unsafe.park() 方法
  2. 检查 _counter ，本情况为 0，这时，获得 _mutex 互斥锁
  3. 线程进入 _cond 条件变量阻塞
  4. 设置 _counter = 0

  1. 调用 Unsafe.unpark(Thread_0) 方法，设置 _counter 为 1
  2. 唤醒 _cond 条件变量中的 Thread_0
  3. Thread_0 恢复运行
  4. 设置 _counter 为 0

9 AQS

9.1 简介

AQS 全称 AbstractQueuedSynchronizer，JUC 下多个同步工具，比如ReentrantLock，CountDownLatch，Semaphore 都是基于此实现

9.2 原理

8：ReentrantLock 原理
加锁流程：当调用ReentrantLock的lock()方法时，首先会调用非公平锁的lock()，使用cas操作把state 设置为1，
如果设置成功，将占有线程设置为当前线程，获取锁成功。 如果设置失败，判断是否是当前线程占有锁，
如果是，state+1 ，获取锁成功 否则，获取锁失败 获取锁失败后，将自己封装成一个Node节点加入到等待队列中去，将自己前驱节点的
状态改为signal，调用park()方法阻塞

解锁流程：当调用ReentrantLock的unlock()方法时，首先会调用非公平锁的release()方法，如果是重入状态，state-1
否则，将持有线程设置为null，state设置为0，唤醒头节点后的第一个节点来竞争锁

await流程：当调用ConditionObject的 await() 方法时，会将当前线程封装成一个Node 节点，状态为Condition，
唤醒头节点后第一个节点

signal流程：当调用ConditionLock的signal方法，会唤醒等待队列中的第一个节点去竞争锁

12: HashMap ConcurrentHashMap
HashMap
1：判断table 是否为null，或者length == 0 ，如果是，就进行resize扩容
2：根据key找到bucket hashcode ^ (hash>>>16) hash & n-1
3: 如果不发生hash冲突，直接插入到bucket 中，如果发生冲突，判读节点是否为treeNode，如果是，直接插入红黑树
否则遍历链表，如果在遍历过程中，key 存在，则覆盖，否则把entry 插入到链表尾部，插入后判读链表长度是否大于阈值，
如果大于阈值，则转换为红黑树
4：插入成功后判断hashmap 容量是否大于阈值，如果大于，就进行resize扩容

1.7 的HashMap(头插法) resize扩容操作可能会导致死链
并发安全的map HashTable，ConcurrentHashMap

ConcurrentHashMap 通过cas 操作和对头节点加锁来保证线程安全
1：初始化Map时，使用cas操作保证只有一个线程进行初始化
2：插入时，如果Bucket 中没有entity，使用cas操作进行插入，如果有entity，锁住头节点进行插入
4：扩容时，锁住头节点进行transfer，transfer 完成的链表或红黑树，将头节点标记为ForwardingNode 表示转移完成

jdk 1.7 segment分段锁，segment继承自ReentrantLock，每个segment 锁住一些bucket
缺点：segment 默认大小为16，不可以改变，非懒惰加载

13 线程池
构造参数： 核心线程数，最大线程数，工作线程存活时间，时间的单位，阻塞队列，线程工程，拒绝策略
线程池状态： Running（创建），Shutdown（调用Shutdown方法），
Stop（调用ShutDownNow），Tidying（所有任务完成，即将进入终结），Terminated（终结）
提交任务，先懒惰创建核心线程，当核心线程达到值定的核心线程数时，再提交任务，将任务加入到阻塞队列，当阻塞队列满之后
再提交任务，创建救急线程执行任务，救急线程+核心线程 不能大于最大线程数，救急线程有存活时间，当达到最大线程数
再提交任务，执行拒绝策略
AbortPolicy 抛异常
DiscardPolicy 直接丢弃
DiscardOldestPolicy 丢弃最早入队的
CallerRunsPolicy 由调用者线程来执行

ShutDown，ShutDownNow
ShutDown 将线程状态改为ShutDown，不会接受新线程，等待所有提交的任务执行完毕
ShutDownNow 将线程池状态改为Stop，不会接受新任务，中断正在执行的任务，队列中的任务作为结果返回
